SIEM Integration

Magertron emits OCSF-aligned audit events designed for SIEM ingestion. Forward them to Splunk, Elastic, Datadog, or Chronicle using the integration pattern that fits your existing observability stack.

✓

Validated end-to-end with Splunk Cloud and Datadog. Audit events ingest with OCSF fields auto-extracted in both platforms. The OCSF specification is supported by every major SIEM (Elastic, Chronicle, Sumo, IBM QRadar) — same events, same schema, same integration shape.

The audit event schema

Magertron writes every audit event to its Postgres database (table audit_events) using the Open Cybersecurity Schema Framework column layout. OCSF is the cross-vendor standard adopted by AWS Security Hub, Splunk, Sumo Logic, IBM QRadar, and many others — meaning your SIEM almost certainly already understands it.

Events follow the OCSF API Activity class (class_uid 6003) and include the standard OCSF base — actor identity, source attribution (IP, user-agent, request ID), severity and outcome, plus correlation identifiers. Magertron-specific context layers on top via OCSF's overflow object.

The schema separates audit emission into two planes:

  • Admin-plane events — orchestrator REST API actions (deploys, role changes, governance changes, license operations).
  • Caller-plane events — per-tool ext_authz authorization decisions when MCP clients invoke tools.

Both planes share the same OCSF base, so a single SIEM ingestion path covers all audit data. SIEM teams typically build separate dashboards per plane — admin-plane for operator-driven changes, caller-plane for runtime tool invocations.

Full column reference and example events are part of the design-partner onboarding.

Why Magertron doesn't push directly to your SIEM

Most enterprise customers run a uniform observability strategy across all their applications — typically a forwarder layer (Splunk Universal Forwarder, Fluent Bit, OpenTelemetry Collector) that ships logs to the SIEM with consistent enrichment, redaction, and rate limiting. Embedding a SIEM-specific HTTP client in the orchestrator would bypass that layer and create a parallel ingestion path that operations teams have to manage separately.

Magertron emits structured audit events. Your existing forwarder picks them up the same way it picks up logs from every other application in your cluster.

Integration patterns

Splunk

Pattern A — Splunk DB Connect (recommended for Splunk shops)

Setup: ~30 minutes Latency: 1–5 min batches Best for: existing Splunk customers

DB Connect queries Magertron's audit table directly and ingests rows on a schedule. Highest fidelity — full OCSF column semantics preserved end-to-end. Recommended approach when SIEM teams already operate Splunk DB Connect for other applications.

  1. Install Splunk DB Connect on a search head (Splunk Cloud or Enterprise)
  2. Add a Postgres identity using a read-only database user we provision
  3. Create a Database Input using the rising-column query template we ship with the integration pack
  4. Set sourcetype to mcp:audit:ocsf and target an index of your choice (we recommend a dedicated index like mcp_audit)

The exact query template, column reference, and indexed-extraction config are part of the design-partner onboarding.

Pattern B — Universal Forwarder + Fluent Bit DaemonSet

Setup: ~1 hour Latency: real-time Best for: Kubernetes-native observability

Fluent Bit runs as a DaemonSet, tails the orchestrator's structured-JSON stdout (which mirrors the audit_events table), and forwards to a Splunk Universal Forwarder Service. The forwarder ships to your Splunk Cloud HEC endpoint.

fluent-bit.conf

[INPUT]
    Name              tail
    Path              /var/log/containers/mcp-orchestratord-*.log
    Parser            json
    Tag               mcp.audit

[FILTER]
    Name              grep
    Match             mcp.audit
    Regex             event_kind audit

[OUTPUT]
    Name              splunk
    Match             mcp.audit
    Host              splunk-uf.splunk.svc.cluster.local
    Port              8088
    Splunk_Token      ${HEC_TOKEN}
    Splunk_Send_Raw   On

Pattern C — Direct HEC push from Magertron (planned for v1.7)

Setup: ~5 minutes (when shipped) Latency: real-time Best for: simple deploys without existing forwarder infra

Native HEC client baked into the orchestrator. Configure with siem.provider: splunk and siem.splunk.hecUrl + a token Secret. Roadmapped for v1.7 — design partners can have it earlier.

Elastic / OpenSearch

Filebeat with the audit_events index template

Setup: ~45 minutes Latency: real-time Best for: Elastic Stack customers

Filebeat tails orchestrator container logs, applies an Elastic Common Schema (ECS) ingest pipeline that maps OCSF fields onto their ECS equivalents (most map 1-to-1 since both standards converged on similar primitives), and ships to your Elasticsearch cluster. We provide the full ingest pipeline configuration on request.

Datadog

Datadog Agent with OCSF source

Setup: ~30 minutes Latency: real-time Best for: Datadog observability customers

The Datadog Agent runs as a DaemonSet, autodiscovers the orchestrator pod, parses JSON logs, and ships to Datadog Logs. Tag with service:mcp-orchestrator and source:ocsf for proper Datadog Cloud SIEM rule application.

Google Chronicle

Chronicle Forwarder with OCSF parser

Setup: ~1 hour Latency: real-time Best for: Google Cloud security teams

Chronicle's recent OCSF-native parsers consume Magertron events directly without custom mapping. Configure the Chronicle Forwarder with the OCSF_API_ACTIVITY log type and point it at the orchestrator's audit log file or Postgres replica.

Common questions

Does Magertron support syslog forwarding for older SIEMs?

Not natively today. Customers running QRadar, ArcSight, or LogRhythm typically use a Fluent Bit or Vector intermediary configured to emit RFC 5424 syslog from the OCSF JSON.

Are caller-plane (per-tool-call) events emitted with the same schema?

Yes. Audit events distinguish admin-plane (orchestrator REST: deploys, role changes) from caller-plane (per-tool ext_authz decisions on MCP traffic) using a dedicated discriminator field. Both share the OCSF base schema. SIEM teams typically build separate dashboards per plane.

How do I correlate audit events with cluster-level activity?

Every event includes a stable cluster identifier and an orchestrator-assigned request correlation ID. The cluster identifier is consistent across orchestrator restarts; the request ID correlates the same logical operation across multiple log lines. Specific field names are documented in the design-partner onboarding pack.

What's the event volume to expect?

Roughly 10–50 admin-plane events per active operator per day (deploys, scales, role changes). Caller-plane events scale with MCP traffic — a 10-server cluster handling ~1k tool calls/day produces ~1k caller events. SIEM ingestion costs are predictable; storage is the dominant variable.
